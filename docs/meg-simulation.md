# Comprehensive Documentation: Biophysical MEG Simulation Engine

## Overview
This simulation engine is a highly customizable, Python-based generator designed to synthesize continuous, biologically plausible Magnetoencephalography (MEG) data. Built entirely around the MNE-Python ecosystem, it is engineered to model the acquisition properties of a 306-channel MEGIN Triux system. 

The primary utility of this engine is twofold: it serves as a rigorous, controlled test harness for validating the Brain Pipes analysis pipeline, and it functions as a highly transparent pedagogical tool for demonstrating concepts like signal averaging, spatial filtering, and inverse modeling to graduate students. By synthesizing everything from 1/f cortical background noise to complex mechanical head drift, the software provides a ground-truth dataset where the precise parameters of the signal and noise are fully known.

---

## Design Goals and Implementation

**1. Python and MNE-Python Native Architecture**
The core engine heavily utilizes `mne.simulation.simulate_raw` and `mne.simulation.SourceSimulator`. Rather than writing a physics engine from scratch, the program leverages MNE's validated forward modeling algorithms to compute the lead fields (the projection of dipole moments from the cortical surface to the MEG sensor array).

**2. FIF Format via Template Substitution**
Generating a completely synthetic `.fif` file from the ground up is structurally brittle due to the complex file headers and acquisition metadata. Instead, the simulator loads a pristine, empty-room or resting-state `.fif` template (from the MNE `sample` dataset). It extracts the `mne.Info` object, explicitly isolates the MEG and relevant physiological/stimulus channels (e.g., stripping out legacy EEG or extra `STI` channels to isolate `STI101`), and overwrites the raw data arrays with the newly simulated sensor timeseries.

**3. BIDS Compliance for Automated Pipelines**
To seamlessly interface with Brain Pipes, the output is strictly standardized using `mne-bids`. The program uses `BIDSPath` and `write_raw_bids` to automatically generate the required directory hierarchy (e.g., `sub-sim01/ses-01/meg/`). It also parses the `STI101` channel to automatically extract event triggers, writing the fully compliant `.json` and `_events.tsv` sidecar files necessary for automated epoching and forked path analysis.

**4. Parameterization via YAML Configuration**
To allow for rapid iteration without altering the Python source code, all simulation variables are abstracted into a configuration file. The program utilizes `ruamel.yaml` to parse this file, enabling dynamic definition of sampling rates, noise amplitudes, mechanical drift vectors, and an arbitrary number of simultaneous, task-evoked sources.

**5. Biophysical and Artifact Simulation Specifications**

* **Real MRI Forward Models:** The simulation relies on the FreeSurfer `oct-6` source space (comprising approximately 7,500 vertices) and a realistic Boundary Element Model (BEM) to ensure the physics of the forward projection represent actual human cortical geometry.
* **AR(1) Background Animation:** To mimic the continuous spontaneous activity of a living brain, the software does not simply add white noise at the sensor level. Instead, a custom Python generator utilizes `scipy.signal.lfilter` to apply a first-order autoregressive AR(1) model to every individual dipole in the source space simultaneously, generating a spatially dense, physiologically realistic 1/f noise floor.
* **Waxing and Waning Oscillations:** The custom generator architecture allows for amplitude-modulated oscillations (such as alpha or beta rhythms) to be superimposed directly onto the AR(1) white noise array prior to filtering, simulating the dynamic state changes of resting-state networks over time.
* **Anatomical ERFs via D-K Atlas:** Task-evoked fields are generated by reading the Desikan-Killiany (D-K) parcellation from the subject's FreeSurfer reconstruction. Users specify anatomical labels (e.g., `transversetemporal-lh`) in the YAML file. The engine generates Hanning-windowed sinusoidal bursts at specified frequencies and SNRs, binds them to the corresponding cortical vertices, and writes the associated synchronization codes directly to the `STI101` channel.
* **External Noise and SSS Validation:** To validate Signal Space Separation (SSS) and tSSS within the pipeline, the simulator is designed to accommodate external noise injection. Because MEG measures vector fields rather than scalar potentials, internal and external magnetic fields are mathematically separable even on a perfect spherical shell. By placing simulated noise dipoles outside the sensor array volume, the engine creates spatially distinct interference patterns that SSS algorithms can isolate and suppress.
* **Intermittent and Continuous Artifacts:** * *Physiological:* EOG (blinks) and ECG (cardiac) artifacts are injected post-projection using MNE's built-in artifact topographies.
    * *Mechanical Movement:* To simulate the massive, multi-channel baseline swings caused by subject movement, the engine synthesizes a continuous 3D translation and rotation trajectory using `scipy.spatial.transform.Rotation`. Because movement alters the physical relationship between the brain and the sensors, the static forward model is discarded. Instead, MNE dynamically recalculates the forward solution at regular intervals along the drift trajectory, perfectly modeling the spatial smearing caused by a moving subject.

**6. Educational and Test Harness Utility**
The combination of BIDS standardization and absolute programmatic control over SNR allows for the creation of graded datasets. Graduate students can be tasked with recovering deeply buried M100s, observing firsthand how mechanical drift destroys signal averaging, and learning how spatial filtering techniques recover the underlying topology.

**7. Support for Forked Path Analysis**
Because the output is deterministic (controlled via a random seed in the YAML) and strictly BIDS-compliant, researchers can run the exact same raw data through multiple branches of the Brain Pipes pipeline simultaneously. This enables robust methodological comparisons—for instance, evaluating the downstream statistical impact of tSSS versus standard Maxwell filtering, or comparing different inverse solvers (e.g., dSPM vs. MNE) on identical biophysical ground truth.

## Configuration Reference: config.yml

The following is a documented sample of the configuration file required to drive the simulation engine. 

    # Simulation Global Parameters
    # ---------------------------------------------------------
    sfreq: 1000.0          # Sampling frequency in Hz
    duration: 60.0         # Total duration of the simulation in seconds
    random_seed: 42        # Seed for deterministic noise generation

    # Background Noise Parameters
    # ---------------------------------------------------------
    noise:
      amplitude_nam: 20.0  # Spontaneous amplitude of the AR(1) source noise in nAm

    # Task-Evoked Responses (ERFs)
    # ---------------------------------------------------------
    # Accepts a list of multiple, simultaneous sources mapped to the D-K atlas
    erfs:
      - name: "auditory_m100"
        labels: 
          - "transversetemporal-lh"
          - "transversetemporal-rh"
        trigger_code: 10
        amplitude_nam: 50.0  # Peak amplitude of the evoked response in nAm
        frequency_hz: 10.0   # Oscillation frequency for the burst
        duration_sec: 0.2    # Length of the burst window
        isi_sec: 2.0         # Inter-stimulus interval (fires every 2 seconds)

    # Physiological and Environmental Artifacts
    # ---------------------------------------------------------
    artifacts:
      eog:
        enable: true
        blink_rate_hz: 0.2   # Simulates 1 blink roughly every 5 seconds
      ecg:
        enable: true
        heart_rate_hz: 1.2   # Simulates a resting heart rate of 72 BPM
      
      # Mechanical head movement triggers dynamic forward model recalculation
      movement:
        enable: true
        # Simulates continuous linear drift over the total duration (in meters)
        # e.g., [0.0, 0.0, -0.01] simulates sliding down 1cm in the Z-axis
        translation_drift_m: [0.005, -0.005, -0.01]
        
        # Simulates continuous rotational drift over the total duration (in degrees)
        rotation_drift_deg: [2.0, 0.0, 0.0] 

    # BIDS Export Configuration
    # ---------------------------------------------------------
    bids:
      subject: "sim01"
      session: "01"
      task: "auditory"
      output_dir: "~/Desktop/simulated_meg_bids"
